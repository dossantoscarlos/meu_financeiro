# Contributing Guide

Obrigado pelo interesse em contribuir.  
Antes de abrir issue ou Pull Request, **leia este documento inteiro**.

Contribuições que não seguirem estas regras podem ser **fechadas sem review**.

---

## Como contribuir

Você pode contribuir de três formas:

- Reportando bugs
- Sugerindo melhorias ou novas funcionalidades
- Enviando Pull Requests com código, testes ou documentação

---

## Regras gerais

- Respeite o escopo do projeto
- Mudanças grandes exigem **issue prévia**
- Código precisa ser **legível, testável e consistente**
- Qualidade > velocidade

---

## Reportando bugs

Antes de abrir uma issue:

1. Verifique se o problema já não foi reportado
2. Use o **ISSUE_TEMPLATE**
3. Forneça passos claros para reprodução

Issues sem reprodução clara **não serão investigadas**.

---

## Sugerindo melhorias ou novas funcionalidades

- Explique o problema real que deseja resolver
- Justifique o impacto da mudança
- Esteja aberto a rejeição ou redesign

Nem toda boa ideia entra no projeto.

---

## Fluxo de desenvolvimento

1. Faça um **fork** do repositório
2. Crie uma branch a partir da branch principal  
   Exemplo: `feat/nome-da-feature` ou `fix/descricao-do-bug`
3. Implemente a mudança
4. Adicione ou atualize testes
5. Garanta que lint e testes passam
6. Abra um Pull Request usando o **PULL_REQUEST_TEMPLATE**

---

## Padrão de código

- Siga o estilo já adotado no projeto
- Não introduza padrões novos sem discussão
- Código deve ser claro, direto e sem “atalhos criativos”

Se o projeto usa formatter ou linter, **eles são obrigatórios**.

---

## Commits

Commits devem ser claros e objetivos.

Exemplos aceitáveis:

- `fix: corrige validação de entrada`
- `feat: adiciona suporte a múltiplos formatos`
- `docs: atualiza instruções de instalação`

Evite:

- `ajustes`
- `fix`
- `update`

---

## Testes

- Toda mudança funcional deve ter teste
- Bug fix sem teste é incompleto
- Se não for possível testar, **explique no PR**

---

## Pull Requests

- PR pequeno e focado é preferível
- PR grande sem issue prévia pode ser fechado
- Explique **o problema**, **a solução** e **o impacto**

O maintainer decide o que entra no projeto.

---

## Code review

- Feedback é técnico, não pessoal
- Ajustes podem ser solicitados
- O maintainer tem a palavra final sobre merge

---

## Compatibilidade e breaking changes

- Mudanças que quebram compatibilidade devem ser explicitadas
- Breaking change sem aviso será rejeitada

---

## Código de Conduta

Ao contribuir, você concorda em seguir o `CODE_OF_CONDUCT.md`.

Comportamento tóxico ou desrespeitoso não será tolerado.

---

## Manutenção

Este projeto é mantido conforme disponibilidade do maintainer.  
Não há SLA de resposta, mas contribuições de qualidade recebem prioridade.

---

## Considerações finais

Se você chegou até aqui e concorda com as regras:

- Seja bem-vindo
- Contribuições bem feitas são valorizadas
- Ruído será ignorado
